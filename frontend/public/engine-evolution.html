ERA 07

The regression cascade · iter A through E

2026-05-24 → 2026-05-26 · 3 days, 11 commits REGRESSION

What went wrong

Day-1 audit found an 80-article "Observer" false-merge (publisher-bridge). The fix shipped as a cohesion-gated mega-cap (MEGA_COHESION_FLOOR=30, articles ≥ 20) plus a new is_headline signal. Within hours is_headline failed to fire anywhere; that triggered iterations B through E that layered adaptive overrides on top of the static thresholds — three-tier corpus-volume scaling for the ranker, two-band adaptive entity-merge IDF (2.0 → 3.0), adaptive anchor IDF (0.70 → 0.80), a 36h recluster window. Each layer was a band-aid for a symptom the previous layer created. By 5/26 the engine had a recluster-only path that processed zero articles (a wipe bug), a Phase 5 trip that shattered any 12-source cluster with multi-desk title diversity, and a rerank writeback that overwrote correct headline flags with strict busy-mode thresholds every single run.

ERA 08

The unwind

2026-05-28 → present · ongoing UNWIND

What shipped

A 9-angle multi-finder code review (4 parallel adversarial agents) surfaced six root-cause regressions from Era 7. One commit unwound them all: the recluster wipe bug got an if not recluster_only guard; Phase 5 cohesion-trip gained a 40-source floor; rerank's corpus_size switched from all-time-DB to a 48h window; both adaptive overrides reverted to static; Phase 5 sub-cap now applies wire-aware collapse before threshold comparison; tabloid keyword matching switched from substring to word-boundary tokens. Validation suite stayed clean (33 CORRECT / 1 ACCEPTABLE / 1 WRONG). The next run lifted max source_count from 3 to 13 and headlines from 0 to firing again — partial recovery. The structural fragmentation in Phase 2 (the len(shared) ≥ 3 gate against shared low-specificity entities like Trump + Iran) remains an open issue.

What the data taught us

Three patterns showed up over and over

1. Symptom-driven fixes accrete dead weight. The Era 7 cascade is the textbook case. Each commit "fixed" the next symptom without going back to find the original cause. By the end, three different adaptive thresholds were all band-aiding the same underlying problem (Phase 5 over-firing) — and unwinding any single one didn't help because the others still tripped.

2. Production-scale calibration drifts silently. The Era 5 thresholds (entity 2.0, jaccard 0.27, anchor 0.70) were calibrated against a 33-fixture validation suite. Production corpus distributions (7K articles, IDF for "Trump" ≈ 2.2 not 5.0) differ. The suite stays green while real-world clustering degrades. Era 4's calm wasn't because the engine was robust — it was because the news cycle happened to give the engine entity overlap it could see.

3. The same knob keeps appearing under different names. similarity_threshold, ENTITY_MERGE_IDF_THRESHOLD, adaptive overrides, MEGA_COHESION_FLOOR, MEGA_COHESION_MIN_SOURCES, the literal 35.0 in the chain-merge-dissolve branch — all are "how lenient is the merge?" If we'd kept a single named scalar with intentional callsites, the regression would have been one diff line not three layered overrides.

Open architectural debts (not in any era's "what shipped")

Union-find ceiling bypass. _would_exceed_ceiling compares clusters[find(i)].source_count + clusters[find(j)].source_count > 120. But union() only updates parent pointers — never accumulates merged sizes back into clusters[root]. Transitive A→B→C→D chains pass each pairwise check while producing a 150-source mega. The Era 5 hard ceiling has been silently bypassed since the day it shipped.

Phase 2 distinguishing-entity gate. Both len(shared) ≥ 3 and distinguishing_shared ≥ 1 assume each cluster carries ≥ 3 named entities. Production reality: many articles mention only 2–3 entities total, and the relevant ones (Trump, Iran, Newsmax) are all in _LOW_SPECIFICITY_ENTITIES. The fallback Jaccard path also requires len(shared) ≥ 3, so it doesn't rescue. Result: real wide-coverage stories where each desk picks a different secondary detail (Hormuz vs. Witkoff vs. ceasefire) end up fragmented across 4-5 sibling clusters.

Two-corpus_size mismatch. main.py passes len(stored_articles) (this run's fetch, 4–13K). rerank.py originally passed len(articles_by_id) (all-time DB, 40K+). The unwind switched rerank to a 48h window. There's still semantic drift between "how much fresh news was there" and "how many clusters are competing." A single, named, run-level corpus signal would remove this ambiguity entirely.
