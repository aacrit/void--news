Simplification plan — what's reverting, and why

The May 15 rewrite added complexity that the production data doesn't reward. This plan unwinds it. Six engine reverts, one perf fix, three new validation fixtures, fifteen execution steps, zero schema rollbacks. Net result: a smaller, faster, more correct engine that does what the March 28 version did while keeping wire-fingerprinting and the 75-source mega cap from the rewrite.

The headline change

Phase 2 entity-merge is restored to its pre-rewrite shape. Phases 2.5, 2.55, 2.6 either collapse into Phase 1 (synonyms) or move behind a default-off flag (anchor). Phase 5 sheds the cohesion-trip and recursive sub-cap and becomes a single soft cap. Everything else stays put.

The six reverts, one by one

Each card explains exactly what's reverting, what code is being deleted or restored, why it solves a real production failure, and what the line-count impact is.

REVERT 01

Phase 2 entity-merge — restore the 2026-03-28 design

REVERT −120 LOC

What's changing

today's gate
score >= 2.0 IDF
AND len(shared) >= 3
AND distinguishing_shared >= 1
AND wire-bridge IDF downweight 0.4×
restored gate
len(shared) >= 3 entities
AND time_spread <= 48h
AND group_size + group_size <= 50 articles
(union-find tracks size correctly)
Why: The current gate requires at least one shared entity that is NOT in _LOW_SPECIFICITY_ENTITIES. Every realistic Trump-Iran / Trump-China / Israel-Gaza story shares only {trump, iran} or similar — all hot-spots, all downweighted. The gate fails on stories the engine should obviously merge. The old gate didn't need entity quality scoring because the 50-article hard cap (with working union-find size tracking) already prevented the chain-merge mega-clusters this complexity was added to stop.

Production evidence

Today's Trump-Iran deal story is fragmented across 5 clusters of sc=6 + 4 + 4 + 3 + 2 = 19 sources. The old engine on the same input would have caught them all in Phase 2's set-intersection. The 233-source AI-deployment chain merge that motivated the rewrite would have been blocked by the old engine's 50-article cap before it could form.

Code-level change

Replace merge_related_clusters() body in pipeline/clustering/story_cluster.py with the version from commit 5402f7d. Drop _LOW_SPECIFICITY_ENTITIES frozenset, _LOW_SPECIFICITY_WEIGHT, distinguishing_shared loop, ENTITY_MERGE_IDF_THRESHOLD, the fallback-Jaccard branch, the broken _would_exceed_ceiling. Restore the union-find with group_size[rb] += group_size[ra] on union.

REVERT 02

Phase 2.55 synonyms → move into Phase 1 normalisation

RELOCATE −80 LOC

What's changing

today
Phase 2.55 — merge clusters that share entities under synonym pairs
{DRC ↔ Congo, UK ↔ Britain, Czechia ↔ Czech Republic, ...}
Runs AFTER Phase 1 vectorisation
simplified
Normalise inside _build_document() before TF-IDF vectorisation:
"Democratic Republic of Congo" → "Congo"
"UK" → "United Kingdom"
Phase 1 vectors already canonical → Phase 2.55 deletable
Why: Phase 1 cosine similarity is computed on TF-IDF vectors of title + 500 words of full text. If those documents are normalised first, the cosine signal between a "DRC" article and a "Congo" article goes up by exactly the right amount automatically. No separate merge pass needed. Keeping the synonym dictionary as a preprocessing step makes the rest of the pipeline simpler without losing any behaviour.

Production evidence

Validation fixture 031-no-regression-republic-of-congo-vs-drc.yaml exists. The simplification rewrites that fixture to test the normalisation path inside Phase 1, not a separate merge phase. Same assertion — Congo and DRC articles cluster together — different mechanism.

Code-level change

Add _SYNONYM_NORMALISATION dict (the existing pair list, flattened to canonical-form mapping). Apply at the top of _build_document() via simple regex substitution. Delete merge_synonym_pairs() function and its call site in cluster_stories().

REVERT 03

Phase 2.5 canonical-pairs — delete entirely

DELETE −150 LOC

What's changing

today
_SUMMIT_PAIRS frozenset of editorial overrides
{trump, xi}, {trump, putin}, {trump, modi}, {starmer, streeting}
When BOTH names appear in both clusters → force merge
simplified
Phase 2 set-intersection picks up {trump, xi, summit, beijing, g20} naturally.
Editorial override list is unnecessary once Phase 2 works.
Why: The original Phase 2 already merged Trump-Xi summit clusters before the rewrite — the entities {trump, xi, summit, beijing} overlap fine, and the simple len(shared) >= 3 gate fires correctly. The summit-pairs editorial override was added because the rewritten Phase 2's distinguishing_shared >= 1 gate was failing on these exact cases. With Phase 2 restored, the override is solving a problem that no longer exists.

Production evidence

The pre-rewrite engine (commit 5402f7d) ran production from 2026-03-28 through 2026-05-14 with NO summit-pair override and successfully consolidated Trump-Xi G20 coverage. The need for the override only appeared after the rewrite.

Code-level change

Delete _SUMMIT_PAIRS, merge_canonical_pairs(), and the call site in cluster_stories(). If a regression appears in a fresh fixture (002-streeting-fragmentation), restore selectively. Otherwise gone.

REVERT 04

Phase 2.6 anchor-entity merge — move behind a default-OFF flag

FLAG −50 LOC (active path)

What's changing

today
Always runs after Phase 2.55.
Five knobs: ANCHOR_IDF_FRACTION_OF_MAX, ANCHOR_MIN_SHARED, ANCHOR_TITLE_JACCARD_FLOOR, plus df cap and IDF cutoff.
simplified
Behind --enable-anchor-merge CLI flag, default OFF.
Available for diagnostic runs but not in the cron path.
Why: Phase 2.6 catches international fragmentation cases (Burkinabé minister, Latvian PM) by merging clusters sharing rare proper nouns like "Siliņa" or "Sheinbaum". The simplified Phase 2 catches MOST of these already (the rare names ARE distinguishing under simple set-intersection). What's left is a small edge-case rescue. Keep the code for when it's needed; don't run it by default.

Production evidence

The pre-rewrite engine ran without an anchor-merge phase and the user remembers it as "worked perfectly." The two months of production before the rewrite show no fragmentation-of-international-news complaints in any session transcript.

Code-level change

Add enable_anchor_merge: bool = False parameter to cluster_stories(). Wrap the existing Phase 2.6 call in if enable_anchor_merge:. CLI flag wired through pipeline/main.py for opt-in runs. Validation fixtures that test anchor merge run with the flag explicitly on.

REVERT 05

Phase 5 mega-cap — single soft-cap branch, no force-split, no cohesion

REVERT −500 LOC

What's changing

today
200-line function. _force_split_cluster(), _cluster_cohesion() scorer, recursive sub-cap, chain-merge-dissolve, shattered detection, mega-cap-shattered fallback. Seven constants.
simplified
14-line function. At source_count >= 75: cap and stamp flag. Done.
No re-split. No cohesion. The 50-article cap in Phase 2 already prevents the chain-merges that motivated all this.
Why: Phase 5 was originally a soft-cap for the rare genuine mega-event (a presidential inauguration, a 9/11-scale disaster). The Era 6/7 elaborations (cohesion-gated trip, force-split, chain-merge-dissolve, recursive sub-cap) were added to compensate for chain-merges that Phase 2 was no longer preventing. With Phase 2 restored to its working state, none of those elaborations have anything to do.

Production evidence

The cohesion-trip path in today's code force-splits any 20-article cluster scoring cohesion below 30 — including legitimate 12-source breaking stories with multi-desk title diversity. The May 28 unwind added a MEGA_COHESION_MIN_SOURCES = 40 floor to narrow the trip, but the underlying mechanism (force-split + shatter + dissolve) is still wrong shape for the problem. Removing it is cleaner than tuning it.

Code-level change

Replace split_mega_clusters() with the 14-line soft-cap version. Delete _force_split_cluster(), _cluster_cohesion(), MEGA_COHESION_* constants, MEGA_SPLIT_* constants, the chain-merge-dissolve branch.

REVERT 06

is_headline — 3-signal formula, no corpus-size adaptive bands

REVERT −80 LOC

What's changing

today
coverage_ok (5+/2tiers OR 8+) AND rank_ok (adaptive 22/35/45 by corpus) AND not mega_capped AND headline_confidence >= conf_floor (30/50/60 adaptive)
Plus corpus_size parameter wired through rerank + main
simplified
is_headline = source_count >= 8
AND headline_rank >= 40
AND not mega_cluster_capped
No corpus_size parameter. No adaptive bands. headline_confidence becomes a derived display signal.
Why: The three-tier corpus-volume scaling was a symptom band-aid for the broken rerank corpus_size (already fixed). With clustering producing realistic source counts again, the simple sc >= 8 AND rank >= 40 gate captures what humans would call "headline" without any need for day-by-day tuning. The complexity removal is large for a small surface.

Production evidence

The Era 4 (April–mid-May) ranker had no is_headline field at all. Top-of-feed selection was driven by headline_rank alone. The DB column stays for the frontend's query consistency, but its value follows the simple rule.

Code-level change

Drop corpus_size parameter from rank_importance(). Remove three-tier band logic at lines 1573-1579 of importance_ranker.py. Replace headline_confidence blend with a simple two-term linear (50% source_count saturating + 50% headline_rank). Remove corpus_articles_window compute in rerank.py.
